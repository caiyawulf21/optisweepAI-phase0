Optisweep AI Support Assistant — Phase 0 Statement of Work (v2)
1. Executive Summary
This document defines the scope, architecture, implementation plan, and delivery objectives for Optisweep AI Support Assistant — Phase 0.
Phase 0 is a focused proof-of-value implementation centered on:

CAT-1: WCS / Service Failure

The objective of Phase 0 is to validate that an AI-assisted troubleshooting system can:


recognize recurring operational failure signatures from natural language symptom descriptions


classify incidents into validated issue categories


select troubleshooting workflows based on confidence and symptom matching


retrieve grounded troubleshooting knowledge and RCA evidence


guide users through detailed role-based operational workflows


escalate intelligently when engineer or infrastructure involvement is required


provide traceable citations and operational context


Phase 0 is intentionally narrow in scope and is not intended to be a production deployment. The purpose is to validate:


operational usefulness


workflow trust


retrieval quality


orchestration design


symptom-driven workflow selection


escalation quality


before broader Phase 1 investment.

2. Phase 0 Goals

2.1 Symptom-Driven Workflow Selection
The system can identify recurring operational failure signatures from user-described symptoms and select the correct troubleshooting workflow based on confidence.
Example symptoms:


AGVs stopped


no active RMS alarms


all tippers heartbeat timeout


hospital tote removal hangs


system appears active but frozen


The user should not need to know:


the root cause


the workflow name


the subsystem owner


the escalation path


The orchestration layer determines the most appropriate workflow using:


symptom extraction


retrieval evidence


issue-category matching


operational signature matching


workflow confidence thresholds



2.2 Grounded Retrieval
The system retrieves relevant historical incidents, RCA findings, troubleshooting guidance, and workflow evidence from manually curated structured records.
All responses must include:


citations


supporting incident references


retrieval confidence indicators



2.3 Role-Based Guided Troubleshooting
The system guides users through:


support-safe troubleshooting actions


engineer-required recovery procedures


infrastructure diagnostics


escalation checkpoints


validation checks


Each workflow step must contain:


detailed operational instructions


required role or permission level


expected outcome


escalation condition


validation checks


stop conditions where applicable



2.4 Deterministic Escalation
The system determines escalation requirements using deterministic rule-based logic.
Escalation conditions include:


unresolved heartbeat recovery


engineer-only actions required


OT hardware alarms present


infrastructure/network indicators


communication recovery failures


recurring incidents in the same shift


remote access unavailable


Escalation domains include:


application


controls


OT networking


infrastructure


database

6. Workflow Model
Workflows are symptom-driven operational runbooks.

6.1 Workflow Naming Convention
Workflow names must describe:


observable symptoms


operational presentation


troubleshooting context


NOT:


case IDs


customer names


assumed root causes


Correct examples:
heartbeat_timeout_no_rms_alarm_v1agvs_stopped_hospital_remove_hangs_v1service_restart_recovery_flow_v1
Incorrect examples:
case_229374_workflowups_haslet_failure_workflow

6.2 Workflow Selection Logic
The orchestration layer selects workflows using:


symptom extraction


retrieval evidence


issue-category matching


confidence thresholds


Workflows are invoked only when:


symptom confidence is sufficiently high


workflow evidence is strong


operational signatures match validated patterns


Otherwise:


retrieval-only mode is used


clarification questions are asked


escalation may be recommended

10. Success Criteria
Phase 0 is successful if the system can:

10.1 Symptom Recognition
Correctly identify recurring CAT-1 operational signatures from natural language troubleshooting input.

10.2 Workflow Selection
Select the correct workflow using:


symptom extraction


confidence scoring


retrieval evidence


without requiring the user to manually select workflows.

10.3 Guided Troubleshooting
Guide users through detailed role-based troubleshooting procedures.

10.4 Escalation
Generate accurate escalation summaries and route incidents to the correct escalation domain.

10.5 Trust
Demonstrate sufficient operational usefulness that support engineers indicate willingness to continue using the system.

12. Final Recommendation
Phase 0 should focus exclusively on validating that:


operational symptom signatures can be recognized


workflows can be selected by confidence


grounded retrieval improves troubleshooting


role-based workflows improve operational consistency


escalation logic reduces ambiguity


support knowledge can be structured into reusable operational runbooks


The objective is not enterprise production readiness.
The objective is to validate:


symptom-driven orchestration


retrieval quality


workflow trust


escalation quality


operational usefulness


before expanding into broader ingestion, automation, graph systems, or ML capabilities.
